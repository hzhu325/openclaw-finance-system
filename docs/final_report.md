# 基于 OpenClaw 的多智能体金融交易系统实现报告

## 1. 项目说明

本次实习任务要求围绕 OpenClaw 搭建一个多智能体金融交易系统。我在实现时没有直接去做实盘交易，而是先完成一个可运行、可检查的本地原型系统。这样做主要是出于两点考虑：一是金融交易涉及真实资金，系统在没有充分回测和人工确认前不应该直接下单；二是实习阶段更重要的是验证多智能体协作、数据处理、策略生成和风控拦截这一整条流程是否能跑通。

当前项目实现的是 simulation-only 版本。系统可以自动生成行情样例或读取 CSV 数据，计算常用技术指标，组织多空观点辩论，输出结构化交易信号，再交给独立风控模块审查。对于 BUY 或 SELL 这类非 HOLD 信号，系统会生成 HITL 人工审批卡片，并保持 `execution_released=false`，也就是不会释放真实执行权限。

## 2. 项目目录

项目主体目录为 `openclaw-finance-system`，主要结构如下：

| 目录或文件 | 说明 |
| --- | --- |
| `agents/` | 四类智能体的角色设定和工具边界 |
| `config/` | 智能体配置、定时任务、消息网关和新闻源配置样例 |
| `src/finance_agents/` | 行情、指标、辩论、策略、风控、HITL、回测等核心代码 |
| `schemas/` | 交易信号 JSON Schema |
| `memory/` | 共享市场事件和人工决策记录 |
| `reports/` | 系统运行后自动生成的报告 |
| `docs/` | 架构说明、运行手册、分析报告和本报告 |
| `dashboard.html` | 本地可视化验收页面 |
| `open_dashboard.bat` | 一键运行和展示脚本 |

整体流程可以概括为：

```text
行情数据 -> 指标计算 -> 多空辩论 -> 交易建议 -> 风控审查 -> 人工审批 -> 回测和报告
```

## 3. 智能体分工

系统中一共设计了四类角色，每个角色都单独放置了 `SOUL.md` 和 `TOOLS.md`，用于说明它的职责、限制和可使用工具。

### 3.1 协调员 / 投资总监

协调员主要负责把一次分析任务串起来。它不直接做交易，而是负责调用分析、交易和风控相关环节，把不同模块的结果汇总成一份可审计的结论。为了降低误操作风险，协调员被限制为不能调用真实券商接口，也不能绕过风控。

### 3.2 研究员 / 分析师

分析师负责读取行情数据和共享市场上下文，并计算 RSI、MACD、均线、波动率、VaR、CVaR、最大回撤等指标。它的输出不是简单给出“买”或“卖”，而是要同时整理多头和空头理由，便于后续辩论。

### 3.3 交易员 / 执行员

交易员在当前版本里只负责生成模拟订单草稿。订单草稿中包含动作、数量、订单类型、限价、止损、止盈和有效期。由于本项目是安全演示版本，交易员不会连接真实账户，也不会真实下单。

### 3.4 风控经理

风控经理独立于策略模块，负责检查交易信号格式、仓位暴露、VaR、CVaR 和最大回撤。如果风险超限，风控模块可以直接拒绝交易提案。即使风险检查通过，只要信号是 BUY 或 SELL，也必须进入人工审批。

## 4. 数据处理

为了让项目可以在没有第三方数据源的情况下稳定验收，我在数据层做了两种输入方式。

第一种是 CSV 输入。只要在 `data/` 目录下放入 `<SYMBOL>_bars.csv`，并提供 `date,open,high,low,close,volume` 六个字段，系统就会优先使用这个文件。

第二种是确定性样例数据。如果没有 CSV 文件，系统会根据标的名称生成一组可复现的模拟行情。这样每次验收时都能得到一致结果，方便检查系统链路，而不会因为外部数据接口不可用导致演示失败。

共享市场信息放在 `memory/market_events.md` 中。当前版本没有真正接入浏览器新闻抓取，但在 `config/news_sources.example.json` 中保留了后续接入 OpenClaw 浏览器自动化的配置入口。

## 5. 策略逻辑

策略模块并不是直接依赖一句自然语言判断，而是先由本地代码计算硬指标，再结合多空辩论结果生成交易信号。当前使用的主要指标包括：

- RSI14
- MACD 和 MACD histogram
- MA20、MA50
- 1 日收益率和 5 日收益率
- 20 日年化波动率
- 60 日最大回撤
- VaR95 和 CVaR95

交易信号会写入 `reports/latest_signal.json`。信号格式受 `schemas/trade_signal.schema.json` 约束，主要包含标的、周期、建议动作、置信度、理由、指标快照、订单草稿、风险参数和是否需要人工审批。这样做的好处是后续模块不需要从一段自由文本中猜测含义，可以直接读取结构化字段。

## 6. 多空辩论机制

文档要求实现“立论 - 交叉质证 - 终极答辩”的三轮对抗性辩论。我在 `src/finance_agents/debate.py` 中实现了这个流程。

第一轮是立论。系统会根据趋势、动量、RSI、收益率和尾部风险分别整理多头理由与空头理由。

第二轮是交叉质证。多头会质疑风险是否已经反映在价格中，空头则会质疑信号是否过度依赖近期走势、是否忽略尾部亏损。

第三轮是终极答辩。双方会给出更谨慎的最终表述，例如只有在风控通过和仓位受控的前提下才考虑交易。

辩论结果会保存到 `reports/latest_debate.md`，同时给策略模块提供一个 net score。这样可以避免只看单一指标，也让最终建议有更清楚的解释来源。

## 7. 风控与 HITL

风控部分是本项目比较重要的一块。系统不允许策略模块自己决定是否放行交易，而是把交易信号交给 `risk.py` 单独审查。

当前风控检查包括：

- 仓位暴露是否超过上限；
- VaR95 是否超过阈值；
- CVaR95 是否超过阈值；
- 60 日最大回撤是否超过阈值；
- 交易信号是否符合 Schema；
- 非 HOLD 信号是否需要人工审批。

HITL 部分由 `hitl.py` 负责。系统会生成人工审批卡片和模拟订单草稿，文件包括：

```text
reports/latest_human_approval_card.json
reports/latest_human_approval_card.md
reports/latest_simulated_order_draft.json
```

在当前实现中，只要是非 HOLD 操作，审批状态就会保持为 `AWAITING_HUMAN`，执行权限保持为 `false`。这说明系统最多给出建议和订单草稿，不会自动把交易发送到真实市场。

## 8. 运行与验证

为了方便验收，我添加了一个一键脚本：

```text
open_dashboard.bat
```

双击后会自动执行以下操作：

```powershell
python -B .\scripts\run_mvp.py --symbol TEST --mode backtest
python -B -m unittest discover -s tests
```

脚本随后会启动本地展示页面。页面上可以看到交易标的、交易建议、风控结论、HITL 状态、核心指标、多空三轮辩论和回测结果。

最近一次测试结果为：

```text
Ran 3 tests
OK
```

这说明核心流程的自动化测试已经通过。

## 10. 实现中的问题和取舍

在实现过程中，有几处地方没有直接做成完整生产版本，而是先采用了更适合实习验收的处理方式。

第一，行情数据没有直接依赖外部 API。因为不同数据源需要账号、Token 或网络环境支持，验收时不一定稳定，所以当前系统优先支持 CSV 文件；如果没有 CSV，则使用 deterministic sample 生成可复现的样例行情。这样可以保证展示和测试结果稳定，但缺点是它不能代表真实市场的实时变化。

第二，非结构化新闻抓取目前只保留了配置入口。项目中提供了 `config/news_sources.example.json`，也通过 `memory/market_events.md` 作为共享上下文入口，但还没有真正接入 OpenClaw 浏览器自动化去定时抓取财经新闻。当前报告里会明确提示外部新闻自动化尚未连接。

第三，HITL 人工审批目前是本地文件形式。系统会生成审批卡片和模拟订单草稿，但没有真正发送到 Slack、Telegram 或企业微信。这是出于安全和配置成本考虑，因为真实消息网关需要额外的机器人 Token 和目标频道配置。当前版本的重点是验证“非 HOLD 信号必须等待人工确认”这个安全逻辑。

第四，回测部分还比较简化。当前回测可以展示初始资金、最终权益、收益率、最大回撤和交易次数，但还没有加入手续费、滑点、停牌、流动性约束等真实交易成本，也没有和买入持有、传统多因子策略做完整横向对比。

这些问题不影响本项目作为原型系统展示完整链路，但如果后续继续完善，需要优先接入真实行情源、新闻自动化、消息网关和更严格的回测基准。

## 11. 总结

本项目完成了一个从数据输入到交易建议、风控审查、人工审批和回测展示的多智能体金融交易系统原型。和单脚本策略相比，这个项目更强调分工和安全边界：分析模块负责证据，策略模块负责生成信号，风控模块负责审查，HITL 模块负责阻断真实执行。

目前系统仍有一些可以继续完善的地方。例如，真实行情接口还没有正式接入，新闻抓取和消息网关也只是配置样例；回测方面还可以加入更多基准策略，例如买入持有、传统多因子策略和单模型决策策略。后续如果继续扩展，可以先接入 YFinance、AKShare 或 Tushare，再把 OpenClaw 浏览器自动化用于市场新闻抓取，最后再考虑纸面交易环境。

总体来看，本项目已经覆盖实习任务要求中的智能体配置、数据流水线、对抗性辩论、风控拦截、HITL 人工审批、自动化调度样例和回测报告。当前版本适合作为教学和研究型原型，不作为真实投资建议。
